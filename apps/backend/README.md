# @our-haven/backend

Phase 2 backend. Node + TypeScript + Fastify + OpenAPI source-of-truth + Postgres (Supabase) + Fly.io. Platform stack per ADR-0010 (supersedes Firebase + GCP from ADR-0004 §§4–7); the Node + TS + Fastify + OpenAPI + Postgres-as-system-of-record + webhook posture from ADR-0004 §§1–3, 8 carries forward unchanged. Pinned to US regions per ADR-0009 (Supabase `us-east-1` + Fly.io `iad`).

## Quickstart

```bash
# from the repo root
npm install
cp apps/backend/.env.example apps/backend/.env
docker compose -f apps/backend/docker-compose.yml up -d   # local Postgres
npm run backend
# → server on http://localhost:8080  (Swagger UI at /docs)
```

For full local Supabase (Auth + Storage + Realtime), install the Supabase CLI and run `supabase start` from the repo root. Otherwise point `SUPABASE_URL` at a hosted dev project.

## Layout

```
src/
  app.ts                Fastify bootstrap + plugin registration + OpenAPI plugin
  server.ts             Process entry; dependency wiring + signal handling
  config/env.ts         Zod-validated env (Supabase + Postgres + queue names)
  db/
    kysely.ts           Postgres connection (Kysely + pg.Pool)
    schema.ts           Kysely Database interface
    migrations/         Kysely migrations (run via `npm run migrate:*`)
  supabase/
    admin.ts            Supabase service-role client (auth.admin, storage)
    storage.ts          Supabase Storage client + signed-upload-URL helper
  jobs/
    queue.ts            pgmq enqueue helper (delayed jobs — Booking expiry, retention)
  plugins/auth.ts       Supabase access token verification (HS256 via jose)
  routes/
    health.ts           GET /v1/healthz + /v1/readyz
    auth.ts             POST /v1/auth/role-claim, /v1/auth/email-otp/*, /v1/auth/step-up/refresh
    providers.ts        POST /v1/providers (provider sign-up)
    uploads.ts          POST /v1/uploads/signed-url
openapi/
  openapi.yaml          Generated source-of-truth spec — DO NOT hand-edit; regenerated by scripts/emit-openapi.ts.
scripts/
  emit-openapi.ts       Boots the app, asks @fastify/swagger for the spec, writes openapi/openapi.yaml.
  check-openapi-drift.ts  CI check: fails if runtime spec ≠ checked-in yaml.
  migrate.ts            Kysely migration runner (up/down/make).
Dockerfile, fly.toml    Fly.io deploy config (region `iad`).
```

## Deep modules

Per ADR-0004 § Consequences, the deep modules (Booking lifecycle, Pricing & Commission, Cancellation, Disintermediation detector, Search ranking, Rating reveal, Verification workflow, Retention planner) live in `packages/domain/` as pure TS — no DB/SDK imports — and get their collaborators injected at the handler layer.

## Region / residency

Production runs on **Fly.io `iad`** (Ashburn, VA) against a **Supabase US-region project** (`us-east-1`). Daily.co rooms stay US. Per ADR-0010 and ADR-0009 (US-national from day one).

## Auth

Routes verify the Supabase access token (HS256 JWT) locally via `jose` with the project's `SUPABASE_JWT_SECRET` — no network call per request. Custom claims (`role`, `kind`, `state`, `caregiver_category` | `specialty`) live in the JWT's `app_metadata` and are written by the backend via the Supabase admin client. Step-up MFA reads `aal === 'aal2'` + the `amr` array and grants a 15-minute window stored in `auth_step_up_grants` for payout-sensitive endpoints.
